Parsing & Source Analysis

Overview

The Parsing & Source Analysis feature is responsible for extracting structured information from source code files across supported languages and frameworks. Its core purpose is to transform raw source files (such as Vue SFCs, JavaScript, or TypeScript modules) into a normalized, analyzable representation (ParsedFile) containing symbols, imports, and documentation artifacts. This enables downstream documentation generation, code navigation, and integration with planning and site-building workflows.

Parsing is a foundational step in the Pipeline: Document Generation & Planning, providing the raw evidence and symbol tables that power DeepDoc's automated documentation. It is also tightly integrated with Vue Component Parsing and Integration for frontend codebases.

Parsing is the first stage in DeepDoc's evidence pipeline. All downstream documentation, planning, and site generation depend on the accuracy and completeness of this step.

For a high-level view of how parsing fits into the overall system, see DeepDoc Architecture & Overview.

Main Workflows

The primary workflow for parsing and source analysis is as follows:

File Discovery & Dispatch The system identifies source files to parse, determines their type by extension, and dispatches to the appropriate parser via parse_file() (deepdoc/parser/registry.py).

Parsing Entry Point The parser entry point (parse_file() in deepdoc/parser/__init__.py) selects the correct parser (e.g., parse_vue() for .vue files).

Language-Specific Parsing For Vue SFCs, parse_vue() (deepdoc/parser/vue_parser.py) extracts the <script> or <script setup> block, determines the language (JS/TS), and delegates to parse_js_ts() (deepdoc/parser/js_ts_parser.py) for symbol extraction.

Vue-Specific Analysis The Vue parser then analyzes Vue-specific constructs (props, emits, slots, etc.) and augments the symbol table.

Template Block Analysis If a <template> block exists, _extract_template_info() extracts component usage and slot information.

Return Structured Representation The parser returns a ParsedFile object containing all extracted symbols, imports, and a raw content preview.

Core Helpers & Business Rules

parse_vue() (deepdoc/parser/vue_parser.py:25)

Purpose: Parses a Vue Single File Component, extracting script/template blocks, analyzing Vue-specific constructs, and returning a structured ParsedFile.

Parameters:

  • path: Path to the .vue file.
  • content: Raw file content.
  • language: Language hint (not always used).

Key Logic & Branches:

  • Calls _extract_script_block() to get script content, language, and setup mode.
  • If no script block is found, still returns a ParsedFile with template/style info.
  • Determines effective language (typescript if lang is ts/tsx, else javascript).
  • Delegates script parsing to parse_js_ts() (deepdoc/parser/js_ts_parser.py:28).
  • Always sets parsed.language = "vue" for clarity.
  • If a component name is detected and not already in symbols, prepends a component symbol.
  • If <script setup>, extracts Composition API constructs via _extract_script_setup_constructs().
  • Else, extracts Options API constructs via _extract_options_api_constructs().
  • Always calls _extract_vue_runtime_features() to detect runtime integrations.
  • If a <template> block exists, calls _extract_template_info() for slots/components.

Example Usage:

Edge Cases:

  • If both <script setup> and <script> exist, <script setup> is preferred.
  • If no script block, still attempts to extract component name and template info.

_extract_script_block() (deepdoc/parser/vue_parser.py:86)

Purpose: Extracts the content, language, and setup status from <script> or <script setup> blocks.

Returns: (script_content, lang, is_setup)

Logic:

  • Uses regex to find all <script> blocks.
  • Determines language from lang attribute.
  • If both <script setup> and <script> exist, prefers <script setup>.
  • Returns empty string if neither is found.

_extract_script_setup_constructs() (deepdoc/parser/vue_parser.py:138)

Purpose: Extracts Composition API constructs from <script setup> blocks.

Logic & Branches:

  • Looks for defineProps, defineEmits, defineExpose, defineModel, defineSlots.
  • For each, extracts variable name, type/fields, and adds a Symbol to the symbol table.
  • Also extracts reactivity primitives (ref, reactive, computed, etc.) via _extract_composition_refs().

_extract_options_api_constructs() (deepdoc/parser/vue_parser.py:300)

Purpose: Extracts Options API constructs from export default { ... } blocks.

Logic & Branches:

  • Looks for props (object or array), emits, and methods.
  • Adds corresponding Symbol entries for each.

_extract_template_info() (deepdoc/parser/vue_parser.py:383)

Purpose: Extracts component usage and slot info from the <template> block.

Logic:

  • Finds child component tags (PascalCase).
  • Finds named slots and default slot.
  • Adds Symbol entries for slots and components.

parse_js_ts() (deepdoc/parser/js_ts_parser.py:28)

Purpose: Parses JavaScript/TypeScript source, extracting symbols and imports.

Logic:

  • Uses tree-sitter if available; otherwise, falls back to regex.
  • Detects React components and hooks by naming convention.
  • Returns a ParsedFile with extracted symbols/imports.

Symbol (deepdoc/parser/base.py:17) and ParsedFile (deepdoc/parser/base.py:35)

Purpose: Data models for representing code symbols and parsed files.

Key Fields:

  • Symbol: name, kind, signature, docstring, fields, props, is_exported, etc.
  • ParsedFile: path, language, symbols, imports, raw_content.

Conditional Branches & Feature Flags

  • Script Block Preference: If both <script setup> and <script> exist, <script setup> is always preferred.
  • Language Detection: If lang is ts or tsx, treats as TypeScript; otherwise, JavaScript.
  • Component Name Extraction: Attempts to extract from name property, defineOptions, or falls back to filename.
  • Vue-Specific Constructs: Only extracted if present; each construct (props, emits, expose, model, slots) is optional.
  • Template Info: Only extracted if <template> block exists.

State Transitions

The parsing process is stateless with respect to persistent data, but the internal state of the ParsedFile and its symbols evolves as each analysis step augments the symbol table.

Integrations Involved

  • JavaScript/TypeScript Parser: The Vue parser delegates script block parsing to parse_js_ts() in deepdoc/parser/js_ts_parser.py.
  • Vue Runtime Detection: The parser detects usage of Vue Router, Pinia, and composables by analyzing imports and function calls.

For how these integrate with the broader documentation pipeline, see Pipeline: Document Generation & Planning.

Configuration & Environment

No environment variables or config flags directly affect the parsing logic in the provided evidence. All behavior is determined by file content and extension.

Edge Cases & Failure Modes

  • Missing Script Block: If a .vue file lacks a <script> block, the parser still attempts to extract component name and template info, returning an empty symbol list if nothing is found.
  • Malformed Blocks: If <script> or <template> blocks are malformed or contain unsupported syntax, extraction may fail silently (returns empty string or partial info).
  • Ambiguous Language: If the lang attribute is missing or unrecognized, defaults to JavaScript.
  • Multiple Script Blocks: If both <script setup> and <script> exist, only <script setup> is parsed.
  • Non-standard Vue Constructs: Only standard Vue constructs are recognized; custom macros or advanced patterns may not be extracted.

The parser relies on regular expressions for block extraction. Highly unconventional formatting or embedded edge cases may cause missed symbols or incomplete analysis.
