Runtime Services & Operations

Overview

The Runtime Services & Operations feature area covers the orchestration, background processing, and operational lifecycle of DeepDoc's chatbot and documentation intelligence services. This includes:

  • The FastAPI-based runtime service for handling chatbot queries, health checks, and CORS.
  • The background indexer responsible for building and refreshing the chatbot's semantic corpora (code, artifacts, docs, relationships).
  • The embedding and LLM provider clients that power both retrieval and answer generation.
  • Operational hooks for incremental and full index sync, as well as backend scaffolding.

These services are the backbone of DeepDoc's interactive documentation experience, enabling real-time, context-grounded answers for developers. For a high-level architectural context, see DeepDoc Architecture & Overview. For details on parsing and planning, see Parsing & Source Analysis and Pipeline: Document Generation & Planning.

This feature is critical for keeping chatbot answers up-to-date and ensuring operational reliability across deployments.

Main Workflows

1. Chatbot Query Handling

When a user submits a question to the chatbot, the following workflow is triggered:

  1. Request Intake: The FastAPI app (create_fastapi_app() in deepdoc/chatbot/service.py) receives a POST to /query with a QueryRequest payload.
  2. Query Expansion: The question is expanded into alternative queries using LLM (if enabled via config).
  3. Embedding: All query variants are embedded in a batch using the embedding client.
  4. Similarity Search: Each corpus (code, artifact, doc, relationship) is searched for relevant chunks using the query embeddings.
  5. Chain Retrieval: Relationship hits are used to pull in related code chunks (e.g., following import graphs).
  6. Reranking: Candidate chunks are reranked using an LLM for precision (if enabled).
  7. Prompt Assembly: The final prompt is built, including selected code, artifacts, docs, and relationships.
  8. LLM Answer Generation: The answer is generated using the chat client, grounded in the retrieved context.
  9. Response: The answer, citations, and doc links are returned to the client.

2. Background Indexing & Refresh

Indexing is performed either as a full sync or incrementally (e.g., via webhook triggers):

  1. Trigger: Indexing is triggered by a CLI command, webhook, or scheduled job.
  2. Corpus Build: Each corpus (code, artifact, doc summary, relationship) is (re)built using chunkers and planners.
  3. Embedding: Chunks are embedded in batches.
  4. Persistence: Chunks and vectors are saved to disk.
  5. Backend Scaffold: The backend is scaffolded if needed.

Incremental sync only rebuilds affected corpora and merges with existing data, minimizing downtime and resource usage.

Core Helpers & Business Rules

1. ChatbotQueryService (deepdoc/chatbot/service.py)

Handles all query logic, including context retrieval, reranking, and prompt assembly.

Key Methods & Logic

  • query(): Orchestrates the full retrieval and answer process.

    • Branching/Feature Flags:
      • retrieval_cfg["query_expansion"]: If true, expands queries via LLM.
      • retrieval_cfg["rerank"]: If true, reranks candidates via LLM.
      • retrieval_cfg["max_prompt_code_chunks"], etc.: Limits prompt size.
    • Error Handling: If any step fails, returns a fallback error response.

  • _expand_query(): Uses LLM to generate alternative queries for better recall.

    • Branch: If query_expansion is disabled, returns the original question only.
    • Error Handling: If LLM call fails, falls back to no expansion.

  • _multi_query_search(): Searches a corpus with multiple query vectors, merges by max score.

    • Branch: If no records, returns empty.
    • Merges: By chunk ID, keeping the highest score per chunk.

  • _chain_retrieve(): Uses relationship chunks (import graphs) to pull in related code files.

    • Logic: For each relationship hit, finds files mentioned in imports not already in code hits, and adds their top code chunks.
    • Score Discount: Chain-retrieved chunks have their score multiplied by 0.8.

  • _rerank(): Uses LLM to rerank candidate chunks for precision.

    • Branch: If rerank is disabled, sorts by evidence priority and score.
    • Candidate Limit: Only reranks up to rerank_candidate_limit candidates.
    • Error Handling: If LLM call fails, falls back to default sort.

  • _build_prompt(): Assembles the final prompt for the LLM, respecting max character limits.

  • _no_context_result(): Returns a fallback answer if no context is found.

2. ChatbotIndexer (deepdoc/chatbot/indexer.py)

Responsible for building and refreshing all chatbot corpora.

  • sync_full(): Rebuilds all corpora from scratch.

    • Calls chunkers for code, artifact, doc summary, and relationships.
    • Embeds all chunks in batches.
    • Saves chunks and vectors.
    • Scaffolds backend if enabled.

  • sync_incremental(): Incrementally updates corpora based on changed/deleted files or doc slugs.

    • Branching:
      • If a corpus is missing or inconsistent, triggers a full rebuild for that corpus.
      • Only processes changed/affected files for efficiency.
    • Merges: Merges new/changed records with existing, removes deleted.

  • _corpus_needs_rebuild(): Checks if a corpus is missing or inconsistent (e.g., missing files, vector count mismatch).

  • chatbot_index_needs_refresh(): Returns True if any corpus needs a rebuild.

3. Provider Clients (deepdoc/chatbot/providers.py)

  • LiteLLMChatClient: Thin wrapper for LLM chat completions.

    • Error Handling: Raises clear errors if LLM is not installed or if API fails.
    • Config: Reads model, temperature, API key, etc., from config.

  • LiteLLMEmbeddingClient: Thin wrapper for embedding API.

    • Batching: Uses batch size from config (default: 1 for Azure, 24 otherwise).
    • Context Window Handling: If a batch is too large, splits or trims and retries.
    • Error Handling: Raises on API errors, with hints for missing keys.

  • build_chat_client() / build_embedding_client(): Factory functions that build the above clients using config.

State Transitions

The main state transitions in this feature revolve around the lifecycle of chatbot corpora and the operational state of the FastAPI service.

  • IndexMissing: No corpora exist, or files are missing.
  • IndexBuilding: Indexer is running (full or incremental).
  • IndexReady: All corpora are up-to-date and ready for queries.
  • IndexStale: Detected changes or inconsistencies; needs refresh.
  • Serving: FastAPI app is live and serving queries.

Integrations Involved

  • LLM Provider (LiteLLM): Used for both query expansion, reranking, and answer generation. See Integration: Anthropic for details on LLM integrations.
  • Embedding Provider (LiteLLM): Used for embedding code, artifact, doc, and relationship chunks for retrieval.
  • Webhook Handlers: Indexer supports incremental syncs triggered by webhooks. See Integration: Webhook Handlers.
  • Frontend/UI: Consumes the /query endpoint for interactive documentation. See Frontend & UI Integration Overview.

Edge Cases & Failure Modes

  • Missing or Stale Index: If no context is found for a query, the chatbot returns a fallback message suggesting re-indexing.
  • LLM/Embedding API Errors: If the LLM or embedding provider fails (e.g., missing API key, context window exceeded), the service logs an error and returns a user-facing error message.
  • Partial Index Rebuilds: If a previous full sync failed, incremental sync will rebuild any missing corpora.
  • CORS Issues: If allowed origins are not configured, frontend requests may fail.
  • Large Prompt Truncation: If prompt size exceeds configured max, context is truncated to fit.
  • Chain Retrieval Overreach: Chain-retrieved code chunks are score-discounted to avoid polluting top results.
  • Embedding Batch Splits: If a batch is too large for the provider, it is split or trimmed and retried.

Running the indexer with inconsistent or missing config may result in partial or unusable corpora. Always verify configuration before deployment.

Constants, Enums & Status Values

There are no explicit enums or status constants in the core runtime code, but several config flags and publication tiers affect behavior:

  • publication_tier: "core", "supporting", or other — affects evidence priority in retrieval.
  • source_kind: "product", "config", "docs", "ops", "tooling", "test", "fixture", "example", "generated" — used for prioritizing evidence.
  • Config flags: query_expansion, rerank, max_prompt_code_chunks, etc. — see Configuration & Environment.
