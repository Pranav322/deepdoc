Frontend Documentation Site Builder

Overview

The Frontend Documentation Site Builder is responsible for generating, scaffolding, and rendering the entire developer documentation frontend for DeepDoc projects. It transforms the AI-generated documentation plan into a fully functional Fumadocs + Next.js site, handling navigation, page trees, static assets, legacy migration, and integration with features like API reference and chatbot. This builder is the bridge between DeepDoc's core document planning pipeline and the interactive, navigable documentation site delivered to end users.

Design & Purpose

The builder is designed for idempotent, reproducible site generation. It takes a DocPlan (from the planning pipeline) and produces a ready-to-serve Fumadocs site, including:

  • Migrating and normalizing legacy Markdown/MDX content
  • Generating a navigation tree and page structure compatible with Fumadocs
  • Scaffolding all required Next.js, TypeScript, and config files
  • Injecting custom UI components (e.g., chatbot, Mermaid diagrams)
  • Ensuring static assets (logos, favicon) are present
  • Cleaning up obsolete artifacts from previous builds

This design ensures that every run produces a clean, up-to-date documentation site, regardless of prior state.

Public Interface

Main API

  • build_fumadocs_from_plan()
    • Parameters:
      • repo_root: Path — Project root directory
      • output_dir: Path — Where docs content is written
      • cfg: dict[str, Any] — DeepDoc config (site, colors, chatbot, etc.)
      • plan: DocPlan — Documentation plan (see Pipeline: Document Generation & Planning)
      • has_openapi: bool — If OpenAPI API reference should be enabled
    • Returns: None
    • Exceptions: Propagates filesystem and serialization errors.

Artifacts Produced

  • site/ — Next.js/Fumadocs app scaffold
  • site/lib/page-tree.generated.ts — Navigation tree
  • site/public/ — Logos, favicon
  • docs/index.mdx — Landing page (if needed)
  • All required config, layout, and component files for Fumadocs

Internal Mechanics

  • Idempotency: All file writes are designed to be repeatable and safe to run multiple times.
  • Legacy Handling: Migration logic ensures that old Docusaurus/Markdown artifacts are normalized to Fumadocs conventions.
  • Page Tree Construction: Uses the navigation structure from DocPlan to build a nested navigation tree, skipping missing or orphaned pages.
  • Config Generation: All config files (package.json, tsconfig.json, etc.) are generated from templates, with project-specific values interpolated.
  • Feature Flags: has_openapi controls whether OpenAPI reference is included. Chatbot enablement is controlled via config and environment variable.
  • Frontmatter Enforcement: Ensures all .mdx files have minimal frontmatter for Fumadocs compatibility.

Call Patterns

The builder is invoked by the main DeepDoc pipeline after planning is complete. See Pipeline: Document Generation & Planning and DeepDoc Architecture & Overview.

Configuration

Environment Variables

VariableRequiredDefaultDescription
NEXT_PUBLIC_DEEPDOC_CHATBOT_BASE_URLNo""Overrides the backend URL for the chatbot panel in the generated frontend.

Config Keys (from cfg)

  • project_name: Used for branding and package naming.
  • site.repo_url: Used for navigation links.
  • site.colors.primary, site.colors.light, site.colors.dark: Used for theming in global.css.
  • chatbot.enabled: Enables/disables chatbot UI integration.

Performance Considerations

  • File I/O: Most operations are file writes/renames; performance is dominated by disk speed and project size.
  • Page Tree Generation: Linear in the number of pages; negligible unless thousands of pages.
  • Legacy Migration: Only runs on first build or when legacy files are present.
  • Idempotency: All steps are safe to repeat; no incremental build logic (full overwrite each time).
  • Hotspots: None known; all operations are batch and non-interactive.
