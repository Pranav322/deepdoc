Pipeline: Document Generation & Planning

This page documents the document generation pipeline for DeepDoc, covering planning, evidence assembly, and output orchestration. It is the central stage that transforms repository source and context into validated, navigable documentation. For a high-level architectural overview, see DeepDoc Architecture & Overview.

Overview

The Document Generation & Planning pipeline is the heart of DeepDoc's documentation system. It orchestrates the transformation of raw repository artifacts (source files, config, endpoints, integrations, and markdown context) into a structured, validated, and cross-linked documentation site.

Inputs:

  • Parsed repository scan (RepoScan from deepdoc/planner_v2.py)
  • Planning configuration (cfg dict)
  • LLM client for prompt-driven planning and generation (LLMClient from deepdoc/llm.py)
  • Source files, artifact files, and integration signals

Outputs:

  • Structured documentation plan (DocPlan from deepdoc/planner_v2.py)
  • Generated documentation pages (MDX files)
  • Navigation structure for Fumadocs site
  • Manifest and ledger for update tracking

The pipeline consists of three main stages:

  1. Planning: Multi-step bucket-based planner assigns files, symbols, and artifacts to documentation buckets.
  2. Generation: Evidence assembler gathers context for each bucket; LLM generates pages using strict prompts.
  3. Validation & Orchestration: Pages are validated, post-processed, and integrated into the site; errors are handled gracefully.

Processing Logic

The pipeline is orchestrated by PipelineV2 (deepdoc/pipeline_v2.py:170), which runs the following stages:

1. Planning Stage

  • Repo Scan: scan_repo() (deepdoc/planner_v2.py:394) scans the repo, extracting file metadata, endpoints, integrations, and context.
  • Phase 2 Enrichment: run_phase2_scans() (deepdoc/planner_v2.py:628) clusters giant files, builds endpoint bundles, discovers integrations/artifacts.
  • Multi-step Planning: plan_docs() (deepdoc/planner_v2.py:1415) runs three LLM-driven steps:
    • CLASSIFY: Categorizes files/artifacts into system, feature, endpoint, integration.
    • PROPOSE: Creates bucket candidates (pages).
    • ASSIGN: Maps files/symbols/artifacts to buckets.
    • AUTO-GEN endpoint_ref: Generates endpoint reference pages for each API endpoint.

Planning Flow

  • Bucket Assignment: Files are assigned to buckets using semantic token matching (_file_semantic_tokens() and _bucket_semantic_tokens() in deepdoc/planner_v2.py).
  • Ownership Refinement: _refine_bucket_ownership() ensures umbrella buckets don't over-claim files.
  • Section/Contract Assignment: _apply_page_contracts() sets required sections and coverage targets for each bucket.

2. Generation Stage

  • Evidence Assembly: EvidenceAssembler (deepdoc/generator_v2.py:90) gathers source context, helper functions, endpoint details, integration context, config/env signals, and database schema for each bucket.
    • Source Context: _build_source_context() includes full source for small files, signatures/excerpts for larger files, and symbol bodies for giant files.
    • Helper Resolution: _build_helper_context() follows imported repo-local helpers.
    • Config Extraction: _build_config_env_context() extracts env var names from source files.
    • Endpoint/Integration Context: _build_endpoints_detail(), _build_integration_context() enrich endpoint/integration buckets.
  • Page Generation: PageGenerator (deepdoc/generator_v2.py:1444) constructs the prompt (using get_prompt_for_bucket() from deepdoc/prompts_v2.py:1429), calls the LLM, and returns markdown.
  • Post-processing: Fixes Mermaid diagrams, file references, and MDX hazards.

Generation Sequence

3. Validation & Orchestration

  • Validation: PageValidator (deepdoc/generator_v2.py:1581) checks:
    • Required sections present (_check_sections())
    • File references exist and are valid (_check_file_refs())
    • No hallucinated paths (_check_hallucinated_paths())
    • Evidence-backed references only (_check_evidence_backed_refs())
    • Mermaid diagrams are syntactically correct
  • Orchestration: BucketGenerationEngine (deepdoc/generator_v2.py:2405) runs generation in parallel, applies degradation fixes for sparse evidence, and updates the manifest for incremental updates.
  • Site Build: Fumadocs site is built from the plan and generated pages.

Error Handling & Recovery

  • LLM Call Failures: _call_with_retry() (deepdoc/generator_v2.py:2703) uses exponential backoff and jitter for transient errors; max retries are capped (MAX_RETRIES = 5).
  • Sparse Evidence: _apply_degradation_fixes() (deepdoc/generator_v2.py:2750) appends missing sections as stubs or removes hallucinated content if validation fails.
  • File Reference Validation: _validate_file_refs() (deepdoc/pipeline_v2.py:1141) strips references to files that don't exist and corrects line numbers.
  • Manifest & Ledger: Manifest tracks file hashes; if files change, only affected pages are regenerated.
  • Dead Letter Handling: If a page cannot be generated after retries, a stub page is emitted (_generate_stub_page()).
Pages with missing required sections or invalid file references are flagged and either fixed automatically or marked for manual review. LLM failures are retried; persistent failures result in stub pages.

Monitoring & Observability

  • Progress Reporting: Uses rich.progress.Progress for real-time feedback during generation.
  • Metrics: Number of files scanned, pages planned, pages generated, pages failed, and timings for each phase are printed via PipelineV2._print_scan() and _print_summary().
  • Manifest Tracking: File hashes are tracked for incremental updates; only stale pages are regenerated.
  • Validation Results: Each page's validation status, missing sections, and warnings are aggregated and summarized.
To debug pipeline failures, check the validation summary and manifest logs. Use the incremental updater to regenerate only affected pages after source changes.
