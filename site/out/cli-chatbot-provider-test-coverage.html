CLI and Chatbot Provider Test Coverage

Overview

This feature documents the test strategy and coverage for DeepDoc's CLI commands and chatbot provider integrations. It ensures that:

  • CLI commands handle environment variables, artifact cleanup, and config preservation correctly.
  • Chatbot embedding clients robustly handle provider-specific batching, context window errors, and text trimming.
  • Chatbot backend and embedding integrations are validated for correct config merging, port assignment, and origin handling.

The business purpose is to guarantee reliable, reproducible CLI operations and chatbot provider interactions, minimizing surprises for developers and ensuring that integrations (e.g., Azure AI, Deepdoc Chat API, Deepdoc Embed API) behave as expected under all conditions.

For architectural context, see DeepDoc Architecture & Overview and Runtime Services & Operations. For broader QA strategy, see Testing & Quality Assurance Overview.

Main Workflows

CLI Environment Handling and Artifact Cleanup

The CLI tests cover:

  • Loading .env files and merging with shell environment variables.
  • Cleaning up DeepDoc artifacts and config files.
  • Preserving config for rebuilds when requested.
  • Overriding config flags via CLI options (e.g., --skip-api, --api).

Workflow Steps:

Autoload .env file When running a CLI command, DeepDoc checks for a .env file in the repo root and loads it unless the variable is already exported.

Clean command removes artifacts The clean command deletes documentation output, DeepDoc config files, and generated directories, but leaves unrelated files untouched.

Generate command with --clean Removes artifacts but preserves .deepdoc.yaml for rebuilds.

Config override via CLI flags --skip-api disables endpoint page generation regardless of config; --api re-enables it.

Chatbot Embedding Client Behavior

The embedding client tests validate:

  • Batch splitting on context window errors (Azure).
  • Trimming oversized texts and retrying.
  • Provider-specific batch size defaults and aggressive trimming.

Workflow Steps:

Embed request with batch Client sends batch of texts to provider (e.g., Azure).

Context window error handling If provider returns context window exceeded error, client splits batch and retries smaller batches.

Trim oversized text If single text exceeds context window, client trims text and retries.

Provider-specific defaults Azure defaults to batch size 1 and more aggressive trim budget.

Participating Endpoints

While the CLI and embedding client tests do not directly hit HTTP endpoints, they validate config and artifact flows that affect endpoint generation and chatbot backend operation. For API endpoints related to chatbot interaction, see API Family: Chatbot.

  • Chatbot Backend Base URL: Used to determine if local backend should be spawned or external URL used.
  • Chatbot Site API Base URL: Used for frontend integration.

For endpoint details, see Integration: Deepdoc Chat API and Integration: Deepdoc Embed API.

Edge Cases & Failure Modes

  • Env var precedence: If a variable is exported in the shell, .env does not override it.
  • Artifact cleanup: Only DeepDoc-managed files are removed; unrelated files are preserved.
  • Batch splitting: Embedding client may retry multiple times if context window errors persist.
  • Text trimming: If text cannot be trimmed below provider limit, embedding fails.
  • Backend spawn: If backend URL is external, local backend is not spawned.
  • Port assignment: Busy ports are skipped; next available port is chosen.
If a provider returns repeated context window errors, embedding may fail after all retries.
Azure provider uses more aggressive trimming and batch splitting than others.
