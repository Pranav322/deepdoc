Database Models & Schema

Overview

DeepDoc uses SQLAlchemy as its primary ORM for modeling and interacting with the database layer. The schema is designed to support code analysis, documentation planning, artifact tracking, and integration discovery. The models are defined in deepdoc/scan_v2.py and are tightly integrated with the document generation and planning pipeline described in DeepDoc Architecture & Overview.

The schema is optimized for:

  • Tracking code symbols and their groupings (clusters)
  • Capturing evidence for API endpoints and integrations
  • Recording discovered integrations and their metadata
  • Cataloging model files, migrations, and schema artifacts

All model definitions and schema discovery logic are implemented in deepdoc/scan_v2.py. There are no separate migration scripts; schema evolution is managed at the code level.

Relationships Summary

The schema is highly compositional, with most relationships being one-to-many (e.g., a GiantFileAnalysis has many SymbolClusters, a DatabaseScan has many ModelFileInfos). There are no many-to-many relationships or foreign key constraints in the traditional RDBMS sense, as the models are primarily used for in-memory analysis and reporting.

Migrations

DeepDoc does not use a traditional migration framework like Alembic. Instead, schema evolution is managed directly in code via the model definitions in deepdoc/scan_v2.py. When the schema changes, you must update the model classes and any downstream consumers.

If you are integrating DeepDoc with a persistent database, you will need to write your own migration scripts or use a tool like Alembic to generate and apply migrations based on the SQLAlchemy models.

For setup and environment configuration, see Setup & Getting Started.

Query Patterns & Performance

  • Query Patterns:
    Most queries are in-memory traversals and aggregations, not SQL queries. For example, endpoint evidence is assembled by traversing the EndpointBundle and EvidenceUnit objects.
  • N+1 Risks:
    There are no N+1 query risks in the default implementation, as all data is loaded and processed in memory.
  • Heavy Joins:
    Not applicable; relationships are managed via Python lists, not SQL joins.
  • Denormalization:
    Some denormalization exists (e.g., storing symbol names and evidence strings redundantly) for performance and simplicity.

If you adapt DeepDoc for persistent storage, review your ORM query patterns to avoid N+1 issues and optimize for batch loading.

Configuration

DeepDoc does not require a database connection by default, as all data is managed in-memory via SQLAlchemy models. If you extend DeepDoc to use a persistent database, you must configure the connection string and pooling in your own integration layer.

For environment setup and configuration, refer to Setup & Getting Started.
