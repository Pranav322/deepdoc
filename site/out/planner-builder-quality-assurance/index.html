Planner and Builder Quality Assurance

Overview

The Planner and Builder Quality Assurance feature area covers DeepDoc's approach to testing and validating its documentation planning (bucket consolidation, decomposition, and navigation shaping) and Fumadocs site generation logic. This ensures that the planner's consolidation algorithms and the builder's site scaffolding workflows are robust, deterministic, and produce high-quality documentation sites.

The planner logic determines how source files and symbols are grouped into documentation "buckets," when to split or merge topics, and how to organize navigation. The builder logic takes these plans and generates a Fumadocs-compatible site scaffold, including handling OpenAPI assets, custom index pages, and special components.

This QA layer is critical for:

  • Ensuring documentation is organized logically and consistently.
  • Preventing accidental merges or splits of important topics (e.g., not merging API endpoint docs).
  • Guaranteeing site builds are reproducible and handle edge cases (e.g., custom index pages, OpenAPI presence/absence).
  • Maintaining compatibility with Fumadocs and DeepDoc's custom MDX conventions.

For a high-level view of DeepDoc's architecture, see DeepDoc Architecture & Overview. For the builder's integration with the frontend, see Frontend Documentation Site Builder. For the overall document generation pipeline, see Pipeline: Document Generation & Planning.

Main Workflows

1. Planner Consolidation & Decomposition

The planner QA tests validate how documentation buckets are split (decomposed) or merged (consolidated) based on file count, symbol count, topic similarity, and special flags (e.g., introduction pages, endpoint refs).

Workflow Steps:

Scan the repository Collect file summaries, symbol tables, and research contexts using _make_scan() (tests/test_planner_consolidation.py:17).

Bucket proposal Create candidate buckets with _make_bucket() (tests/test_planner_consolidation.py:43), including section, title, and owned files.

Decomposition check For each bucket, call _should_decompose() (tests/test_planner_consolidation.py) to decide if it should be split. This uses file count, symbol count, and "giant file" detection.

Consolidation check Call _consolidate_similar_buckets() (tests/test_planner_consolidation.py) to merge highly similar buckets, unless they are introduction pages or endpoint refs.

Navigation structure update Clean up the navigation structure to remove dangling slugs and ensure merged buckets are reflected.

2. Fumadocs Site Generation

The builder QA tests validate the end-to-end generation of the Fumadocs site scaffold, including handling of OpenAPI assets, custom index pages, and special MDX/diagram normalization.

Workflow Steps:

Prepare plan and output directory Use make_plan() and make_bucket() to assemble a plan, and create the output directory.

Build site scaffold Call build_fumadocs_from_plan() (tests/test_fumadocs_builder.py) with the repo root, output dir, config, plan, and OpenAPI presence.

Handle OpenAPI assets If OpenAPI is present, call stage_openapi_assets() to generate the manifest and API route scaffolds.

Normalize MDX and diagrams Use helpers like escape_mdx_route_params(), escape_mdx_text_hazards(), and _fix_mermaid_diagram() to sanitize content.

Emit site files Write .mdx pages, package.json, and site components.

Participating Endpoints

No direct HTTP API endpoints are exercised in these planner and builder QA tests. All logic is internal to the planning and site generation pipeline. For endpoints related to chatbot or OpenAPI, see API Family: Chatbot.

Core Helpers & Business Rules

_should_decompose() (tests/test_planner_consolidation.py)

  • Purpose: Determines if a documentation bucket should be split into smaller topics.
  • Parameters: bucket, scan, threshold
  • Business Rules:
    • File count: Buckets with fewer than 7 files do not decompose (test_should_decompose_below_new_threshold()).
    • Symbol count: Buckets require 40+ symbols AND 5+ files to trigger symbol-based decomposition (test_should_decompose_symbol_threshold_raised()).
    • Giant file: Any bucket containing a "giant file" triggers decomposition, regardless of file count (test_should_decompose_giant_file_still_triggers()).
    • Endpoint refs: Buckets marked with generation_hints={"is_endpoint_ref": True} are never decomposed (test_should_decompose_skips_endpoint_ref()).

Example usage:

_consolidate_similar_buckets() (tests/test_planner_consolidation.py)

  • Purpose: Merges highly similar buckets in the same section, unless forbidden by business rules.
  • Parameters: plan, cfg
  • Business Rules:
    • Similarity threshold: Only merges buckets with similarity above consolidation_similarity_threshold.
    • Section isolation: Buckets in different sections are never merged (test_consolidate_preserves_different_sections()).
    • Introduction/overview pages: Buckets with generation_hints={"is_introduction_page": True} are never merged (test_consolidate_skips_introduction_pages()).
    • Endpoint refs: Buckets with generation_hints={"is_endpoint_ref": True} are never merged (test_consolidate_skips_endpoint_refs()).
    • Chain merges: If A merges into B and B merges into C, A's files end up in C (test_consolidate_chain_merges()).
    • Nav structure cleanup: After merging, navigation is cleaned to remove dangling slugs (test_consolidate_nav_structure_cleaned()).
    • Section/diagram union: Merged buckets union their required_sections and required_diagrams (test_consolidate_merges_required_sections_and_diagrams()).

Example usage:

build_fumadocs_from_plan() (tests/test_fumadocs_builder.py)

  • Purpose: Generates the Fumadocs site scaffold from a plan.
  • Parameters: repo_root, output_dir, config, plan, has_openapi
  • Business Rules:
    • Custom index preservation: If a handwritten index.mdx exists without frontmatter, it is preserved and frontmatter is injected (test_build_fumadocs_preserves_handwritten_index_without_frontmatter()).
    • OpenAPI handling: If has_openapi=False, API route scaffolds and OpenAPI components are omitted (test_build_fumadocs_without_openapi_omits_api_route_scaffold()).
    • Staged OpenAPI operations: If OpenAPI is present but no endpoint pages exist in the plan, staged operations are surfaced as a fallback (test_build_fumadocs_surfaces_staged_openapi_operations_when_plan_has_no_endpoint_pages()).

Example usage:

stage_openapi_assets() (tests/test_fumadocs_builder.py)

  • Purpose: Processes OpenAPI specs and generates a manifest for the site.
  • Business Rules:
    • Slug shape: Endpoint slugs are normalized (e.g., /orders/{id} → get-orders-id) (test_stage_openapi_assets_uses_endpoint_ref_slug_shape()).
    • Server origin stripping: Manifest paths strip server origins (e.g., http://localhost:3000/health → /health) (test_stage_openapi_assets_strips_server_origin_from_manifest_paths()).

State Transitions

The planner's state transitions revolve around the bucket set and navigation structure. Buckets may be split (decomposed), merged (consolidated), or left unchanged, with the navigation structure updated accordingly.

Integrations Involved

  • OpenAPI Spec Handling: The builder logic integrates with OpenAPI spec files to generate API operation manifests and route scaffolds. See Pipeline: Document Generation & Planning.
  • Frontend Site Builder: The output is consumed by the Fumadocs frontend. See Frontend Documentation Site Builder.

Configuration & Environment

  • Consolidation Similarity Threshold: The planner's merging behavior is gated by the consolidation_similarity_threshold config parameter. Lower values make merges more aggressive; higher values make them more conservative.
  • Decomposition Threshold: The default file count threshold for decomposition is 7 files; symbol-based decomposition requires 40+ symbols and 5+ files.
  • OpenAPI Detection: The presence of OpenAPI is detected via .deepdoc/scan_cache.json (scripts/refresh_generated_sites.py:17).

Edge Cases & Failure Modes

  • Dangling Navigation Slugs: After merges, the navigation structure may reference slugs that no longer exist. The planner cleans these up (test_consolidate_nav_structure_cleaned()).
  • Chain Merges: Multiple merges in sequence are supported and tested (test_consolidate_chain_merges()).
  • Giant Files: Buckets with a single "giant file" are always decomposed, even if file count is low.
  • Introduction/Endpoint Ref Immunity: Introduction and endpoint ref buckets are immune to both decomposition and consolidation, preventing accidental loss of key docs.
  • Custom Index Handling: Handwritten index.mdx files are preserved and annotated, not overwritten.
  • OpenAPI Fallback: If OpenAPI is present but no endpoint pages are in the plan, staged operations are surfaced as a fallback navigation group.
  • Omitted API Scaffolds: If OpenAPI is absent, all related API route scaffolds and components are omitted from the site build.
If the planner or builder logic is changed, these edge cases must be re-tested to avoid regressions in doc structure or site generation.

Constants, Enums & Status Values

  • consolidation_similarity_threshold: Configurable float (e.g., 0.30–0.95) controlling how aggressively buckets are merged.
  • Decomposition thresholds: File count (default 7), symbol count (40+), and file count for symbol-based decomposition (5+).
  • generation_hints: Dict with keys like is_introduction_page, is_endpoint_ref, prompt_style — used to gate planner logic.
