Integration: Webhook Handlers

What This Integration Does

Webhook handlers in DeepDoc enable event-driven integrations with external systems, allowing DeepDoc to react to changes, synchronize data, and trigger downstream workflows. These handlers are essential for:

  • Incremental documentation updates: When code, artifacts, or documentation change, webhook events can trigger targeted re-indexing or regeneration.
  • External notifications: DeepDoc can notify or be notified by other systems (CI/CD, code hosts, artifact stores) via webhooks.
  • Custom integrations: Teams can extend DeepDoc to interoperate with internal tools or third-party services using webhook patterns.

This integration is foundational for keeping documentation and knowledge bases in sync with rapidly changing source repositories and external systems.

Where It Enters the Codebase

Webhook handler logic and integration points are distributed across several core modules:

  • Indexing and Corpus Refresh:

    • ChatbotIndexer.sync_incremental() (deepdoc/chatbot/indexer.py:60)
      Handles incremental updates triggered by file changes, which can be initiated by webhook events.
    • chatbot_index_needs_refresh() (deepdoc/chatbot/indexer.py:194)
      Determines if a webhook-triggered event requires a full or partial corpus rebuild.

  • Provider and Client Abstractions:

    • LiteLLMChatClient and LiteLLMEmbeddingClient (deepdoc/chatbot/providers.py:12, deepdoc/chatbot/providers.py:51)
      While not direct webhook handlers, these classes are invoked as part of event-driven workflows, e.g., embedding new content after a webhook-triggered update.

  • Configuration and Environment:

    • get_chatbot_cfg() and related helpers (deepdoc/chatbot/settings.py:77)
      Used to resolve webhook-related configuration, such as allowed origins and API keys.

  • CLI Entrypoints:

    • update() (deepdoc/cli.py:362)
      The CLI command for incremental updates, which can be triggered by webhook events from CI/CD or external systems.

Webhook handler patterns are not always explicit HTTP endpoints in this codebase; many are event-driven functions that can be invoked by webhook events or background jobs.

Participating Features & Endpoints

Webhook-driven workflows are integral to several DeepDoc features:

  • Incremental Documentation Update
    CLI Commands & Usage — The deepdoc update command supports incremental refreshes, which can be triggered by webhook events (e.g., from a GitHub webhook or CI pipeline).

  • Chatbot Corpus Sync
    Runtime Services & Operations — The chatbot indexer (ChatbotIndexer) supports both full and incremental syncs, which can be triggered by webhook events indicating code or artifact changes.

  • Frontend & UI Integration
    Frontend & UI Integration Overview — Webhook-driven updates ensure that the documentation site and chatbot UI reflect the latest source-of-truth.

  • Deepdoc Chat API
    Integration: Deepdoc Chat API — Webhook events can trigger chat API re-indexing or refreshes.

Retry, Reconciliation & Failure Handling

Webhook-driven flows in DeepDoc are designed for idempotency and resilience:

  • Corpus Rebuild on Failure:
    In ChatbotIndexer.sync_incremental() (deepdoc/chatbot/indexer.py:60), if a previous full sync failed or a corpus is missing/inconsistent, the handler triggers a full rebuild for that corpus:

  • Embedding Retry Logic:
    The embedding client (LiteLLMEmbeddingClient.embed(), deepdoc/chatbot/providers.py:60) handles context window errors by splitting batches or trimming text, and retries as needed.

  • Error Propagation:
    If embedding or chat completion fails, exceptions are raised with actionable hints (e.g., missing API key, model error). These are surfaced to the CLI or API caller.

  • No Explicit Circuit Breaker:
    There is no explicit circuit breaker or exponential backoff in the provided code, but the design ensures that failed or partial updates do not corrupt the corpus and can be retried safely.

Operational Gotchas

  • Batch Size Limits:
    Embedding batch size is provider-dependent (batch_size in config, default 1 for Azure, 24 otherwise). Large payloads may be split automatically.

  • CORS and Origins:
    If running the backend locally, ensure DEEPDOC_CHATBOT_PREVIEW_PORT is set so that the frontend can communicate with the backend during preview.

  • Partial Syncs:
    If a webhook triggers an update for a file that is not tracked or already deleted, the handler will skip it gracefully.

  • Embedding Model Identity:
    The embedding metadata includes the model identity (provider, model, base_url, api_version) for traceability.

  • No Built-in Rate Limiting:
    The code does not implement explicit rate limiting for webhook-triggered updates; ensure your external systems do not overwhelm the backend with events.

If the embedding provider returns a "context window" or "maximum context length" error, the system will attempt to split or trim the input and retry. However, repeated failures may require manual intervention.
