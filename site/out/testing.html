Testing & Quality Assurance Overview

Overview

DeepDoc's Testing & Quality Assurance system ensures that all architectural components, integrations, and framework support features are robust, reliable, and maintainable. This layer covers automated tests for framework detection, planner and builder workflows, CLI and chatbot provider integrations, and normalization routines for documentation artifacts. It is foundational to the project's CI/CD pipeline and guarantees that every supported framework, integration, and site generation feature is validated against real-world scenarios.

Testing is tightly integrated with the overall architecture (DeepDoc Architecture & Overview), and directly supports the following domains:

  • Multi-framework support (Framework Support Testing Strategy)
  • Planner and builder QA (Planner and Builder Quality Assurance)
  • CLI and chatbot provider coverage (CLI and Chatbot Provider Test Coverage)
Testing is not isolated — it is cross-cutting, touching every major subsystem and integration, including Integration: Webhook Handlers, Pipeline: Document Generation & Planning, and Frontend Documentation Site Builder.

Architecture & Design

Testing & QA is designed as a cross-cutting layer, validating every major architectural concern. The system leverages:

  • Integration tests for framework detection and repo scanning
  • Unit tests for planner, builder, and normalization routines
  • Mocking and monkeypatching for provider and embedding client behaviors
  • Config-driven test scenarios for site generation and OpenAPI asset staging

Mermaid Diagram: Test Flow

Design Patterns:

  • Fixture-based repo simulation: Tests simulate real project structures (Django, Express, Falcon, Vue) using temporary directories and files.
  • Monkeypatching: Used in tests/test_chatbot_providers.py to simulate provider errors and batch logic.
  • Config-driven planning: Builder tests use config objects to drive site generation scenarios.

Configuration

Testing & QA is primarily driven by:

  • Config objects: e.g., DEFAULT_CONFIG (deepdoc.config) used in repo scanning tests.
  • Provider settings: e.g., Azure embedding client config (provider, model, batch_size) in LiteLLMEmbeddingClient (deepdoc.chatbot.providers.py).
  • Site generation config: Passed to build_fumadocs_from_plan() (deepdoc/site/fumadocs_builder_v2.py).
All test scenarios are self-contained; environment variables are not directly required for test execution. However, production config knobs are validated in Setup & Getting Started.

How Other Components Use This

Testing & QA is foundational for:

  • Framework support: Validates endpoint detection logic (Framework Support Testing Strategy)
  • Planner & builder: Ensures site generation and planning (Planner and Builder Quality Assurance)
  • CLI & chatbot providers: Tests embedding client error handling and batch logic (CLI and Chatbot Provider Test Coverage)
  • Webhook integrations: Validates integration points (Integration: Webhook Handlers)

Edge Cases & Failure Modes

  • Context window errors: Azure embedding client tests simulate context window exceeded errors and validate batch splitting and aggressive trimming. If a batch is too large, the client retries with smaller batches or truncated text.

    • Example: _FakeContextWindowError triggers batch splitting in test_embedding_client_splits_batches_on_context_window_error() (tests/test_chatbot_providers.py:12)
    • Trimming logic ensures text ends with "... [truncated for embedding]".

  • Commented routes: Falcon route detection ignores commented-out routes, ensuring only active endpoints are surfaced.

    • Example: test_scan_repo_ignores_commented_falcon_routes() (tests/test_framework_support.py:371)
    • Gotcha: Only uncommented routes are detected; commented routes are skipped.

  • Multiline imports and nested includes: Django tests validate that nested includes and multiline imports are correctly resolved, ensuring all endpoints are detected even across complex project structures.

  • Normalization hazards: Builder tests ensure that code fences, HTML blocks, and route parameters are correctly normalized and escaped, preventing rendering errors in the documentation site.

Tests rely on simulated repo structures. If the test setup does not match real-world project layouts, detection logic may miss endpoints or signals.
