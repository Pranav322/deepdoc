Integration: Deepdoc Chat API

What This Integration Does

The Deepdoc Chat API powers conversational features in DeepDoc, enabling users to ask questions about their codebase and receive context-aware answers, code citations, artifact references, and documentation links. This integration provides the backend for the chatbot UI, handling natural language queries, retrieving relevant code/doc chunks, and generating answers using LLMs.

Business Role:

  • Enables interactive documentation and code Q&A for end-users.
  • Supports code search, artifact lookup, and documentation linking via chat.
  • Integrates with LLM providers (Anthropic, OpenAI, Azure, etc.) for answer generation.

The Deepdoc Chat API is a core part of DeepDoc's interactive documentation experience. It is not a third-party SaaS, but an internal API scaffolded and optionally run as a local backend.

Where It Enters the Codebase

The Deepdoc Chat API is scaffolded, configured, and started by several key modules:

  • Backend Scaffold & API Types:

    • scaffold_chatbot_backend() (deepdoc/chatbot/scaffold.py:11): Generates the FastAPI backend, schemas, and settings for the chatbot.
    • _app_py() (deepdoc/chatbot/scaffold.py:30): Produces the FastAPI app entrypoint, including error fallback logic.
    • _schemas_py() (deepdoc/chatbot/scaffold.py:68): Defines request/response Pydantic models for the chat API.

  • Configuration & Settings:

    • chatbot_enabled(), chatbot_site_api_base_url(), chatbot_backend_base_url() (deepdoc/chatbot/settings.py:81,97,102): Control whether the chatbot is enabled, and resolve the backend URL.
    • DEFAULT_CHATBOT_CONFIG (deepdoc/chatbot/settings.py:13): Default config structure, including API keys and backend URLs.

  • CLI Integration & Startup:

    • _start_chatbot_backend() (deepdoc/cli.py:889): Scaffolds and launches the chatbot backend as a subprocess during local preview (deepdoc serve).
    • CLI commands: init, generate, serve, clean (deepdoc/cli.py): All support chatbot-related flags and state.

  • Config Loading:

    • load_config() (deepdoc/config.py:203): Loads .deepdoc.yaml config, including chatbot settings.

Frontend/consumer:

  • The frontend UI (see Frontend & UI Integration Overview) communicates with the Chat API at the resolved base URL.

Participating Features & Endpoints

The following features and endpoints use the Deepdoc Chat API:

  • Chatbot Query Endpoint:
    • /query (POST): Accepts user questions and chat history, returns answers with citations and links.
    • /health (GET): Health check endpoint for readiness/liveness.

See API Family: Chatbot for full endpoint documentation.

Auth & Configuration

API Keys & Environment Variables

The Chat API requires API keys for LLM and embedding providers, set via environment variables:

VariablePurposeRequiredDefault
DEEPDOC_CHAT_API_KEYLLM provider API keyYesNone
DEEPDOC_EMBED_API_KEYEmbedding provider API keyYesNone
DEEPDOC_CHATBOT_PREVIEW_PORTFrontend port for CORS previewNoNone

How to set up:

Install dependencies Ensure you have the chatbot extras installed:


pip install deepdoc[chatbot]

Configure environment Copy chatbot_backend/.env.example to .env and fill in your API keys:

DEEPDOC_CHAT_API_KEY=sk-...
DEEPDOC_EMBED_API_KEY=sk-...

Set backend base URL In .deepdoc.yaml, set chatbot.backend.base_url to your backend's URL (or leave blank for local).

Start the backend Run deepdoc serve or start uvicorn chatbot_backend.app:app manually.

Config keys:
See DEFAULT_CHATBOT_CONFIG in deepdoc/chatbot/settings.py:13 for all options.

For full setup instructions, see Setup & Getting Started.

Retry, Reconciliation & Failure Handling

Startup Failure

If the backend fails to start (e.g., missing config, invalid API key), the fallback app in _app_py() (deepdoc/chatbot/scaffold.py:30) exposes:

  • /health: Returns {"status": "error", "detail": "<error message>"}.
  • /query: Always returns HTTP 503 with {"error": "startup_failed", "detail": "<error message>"}.

Branching logic:

  • If create_fastapi_app() fails, the fallback FastAPI app is used, ensuring the frontend receives a clear error instead of connection refused.

Error Handling

  • LLM/Embedding errors: If the LLM or embedding provider is unreachable or misconfigured, errors are surfaced in the /health endpoint and as 503 errors on /query.
  • CORS: The backend uses permissive CORS settings (allow_origins=["*"]) in fallback mode to ensure error messages reach the frontend.

No Automatic Retry

There is no built-in retry or circuit breaker logic in the scaffolded backend. Errors are surfaced immediately to the frontend for user awareness.

If the backend is down or misconfigured, chat features will be unavailable until resolved. Monitor /health for status.

Operational Gotchas

  • CORS: During local development, CORS is set to allow all origins. In production, restrict allowed_origins in .deepdoc.yaml for security.
  • Port Conflicts: The backend defaults to port 8001 or a hash-based port if running multiple instances. If the preferred port is busy, a nearby available port is chosen (deepdoc/cli.py:_find_available_loopback_port).
  • API Key Leaks: Never commit .env files with real API keys.
  • Payload Size: The default config allows up to 200,000 prompt characters (max_prompt_chars). Large queries may be truncated.
  • Provider Limits: LLM and embedding providers (Anthropic, OpenAI, Azure) may enforce their own rate limits and quotas. Errors from these providers are surfaced as 503s.
  • Versioning: The backend is scaffolded per-repo. If you upgrade DeepDoc, re-run deepdoc generate to refresh the backend scaffold.

Use the /health endpoint to monitor backend status and error details.
