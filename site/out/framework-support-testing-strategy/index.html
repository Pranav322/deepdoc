Framework Support Testing Strategy

Overview

DeepDoc's Framework Support Testing Strategy ensures robust, accurate detection and analysis of API endpoints and components across multiple backend and frontend frameworks—including Django, Express, Fastify, Falcon, and Vue. This feature is critical for DeepDoc's promise of automated, framework-agnostic documentation generation, enabling teams to trust that their APIs and UI components are correctly discovered and described regardless of stack.

The test suite validates DeepDoc's ability to:

  • Detect and resolve routes, handlers, and viewsets in complex, real-world codebases.
  • Traverse multi-file, multi-module project structures.
  • Correctly handle framework-specific conventions, wrappers, and edge cases.

This strategy underpins the reliability of DeepDoc's core pipeline (DeepDoc Architecture & Overview), and is foundational for downstream features such as Pipeline: Document Generation & Planning and Frontend & UI Integration Overview.

For a high-level view of DeepDoc's overall testing and QA approach, see Testing & Quality Assurance Overview.

Main Workflows

The primary workflow for framework support testing is end-to-end validation of route and component detection across supported frameworks. The process is as follows:

Prepare Test Project Structure Each test creates a synthetic or real project structure (e.g., Django app, Express API, Vue SPA) using temporary directories and files.

Write Framework-Specific Source Files Test code writes representative source files (e.g., urls.py, server.js, .vue files) with realistic routes, handlers, and components.

Invoke DeepDoc Scanning The test calls scan_repo() (deepdoc/planner_v2.py) or detect_endpoints() (deepdoc/parser/api_detector.py) to analyze the project.

Assert Detected Artifacts The test asserts that the expected endpoints, handlers, methods, and components are present in the scan results, including correct path/method/handler mapping.

Edge Case & Regression Checks Tests cover edge cases such as nested includes, sync/async wrappers, commented routes, and multi-file mount chains.

Main Flow Diagram

  • scan_repo() and detect_endpoints() are the main entry points for analysis.
  • Route/component resolution may involve cross-file import analysis, handled by helpers in deepdoc/parser/routes/repo_resolver.py.

For details on the document generation pipeline that consumes these results, see Pipeline: Document Generation & Planning.

Core Helpers & Business Rules

1. detect_django() (deepdoc/parser/routes/django.py)

  • Purpose: Detects Django routes, infers HTTP methods, resolves includes and routers.
  • Key Logic:
    • Scans for urlpatterns and path()/re_path() calls.
    • Handles wrappers (e.g., csrf_exempt, sync_to_async).
    • Infers methods from decorators (@api_view), function bodies (request.method), and class bases (e.g., ListView).
    • Resolves DRF routers and ViewSet actions.
  • Branching:
    • If a route handler is an include, recursively resolves child endpoints.
    • If a handler is a class view, infers methods from class definition.
    • If a handler is a function, infers methods from decorators or body.
  • Example:
  • Edge Cases:
    • Handles nested includes, settings-based prefixes, and sync/async wrappers.

2. detect_express() (deepdoc/parser/routes/express.py)

  • Purpose: Detects Express.js routes, including chained and mounted routers.
  • Key Logic:
    • Scans for .get(), .post(), .route(), and .use() calls.
    • Resolves prefixes from router mounts, including multi-file chains.
    • Extracts handler and middleware names.
  • Branching:
    • Skips Fastify objects.
    • Handles both direct and chained route definitions.
  • Example:
  • Edge Cases:
    • Handles routers mounted across multiple files.

3. detect_fastify() (deepdoc/parser/routes/fastify.py)

  • Purpose: Detects Fastify routes, plugin prefixes, and schema definitions.
  • Key Logic:
    • Scans for .get(), .route(), and plugin registration.
    • Extracts request/response schemas.
  • Branching:
    • Only processes objects recognized as Fastify instances or plugin aliases.
  • Example:
  • Edge Cases:
    • Handles plugin prefixes and schema extraction.

4. detect_falcon() (deepdoc/parser/routes/falcon.py)

  • Purpose: Detects Falcon routes and maps resource classes to HTTP methods.
  • Key Logic:
    • Scans for add_route() calls.
    • Extracts path, resource reference, and finds on_* methods in resource class.
    • Skips commented-out routes.
  • Branching:
    • If resource class defines on_* methods, emits endpoint for each.
    • If not, emits generic endpoint with method "ANY".
  • Example:
  • Edge Cases:
    • Ignores commented routes, resolves constants in path expressions.

5. resolve_repo_endpoints() (deepdoc/parser/routes/repo_resolver.py)

  • Purpose: Cross-file/module endpoint normalization and resolution.
  • Key Logic:
    • Builds JS and Python import indices.
    • Resolves handler files, route files, and prefixes across modules.
    • Handles Django, Express, and Falcon specifics.
  • Branching:
    • If endpoint is Express, uses _resolve_express_endpoint.
    • If Falcon, uses _resolve_falcon_endpoint.
    • If Django, triggers _resolve_django_repo_endpoints for full URL tree.
  • Example:
  • Edge Cases:
    • Handles cyclic imports, missing files, and ambiguous handler references.

State Transitions

While the feature does not manage persistent state, it does transform the state of detected endpoints/components as it parses, resolves, and normalizes them.

Endpoint Detection State Diagram

  • Unparsed: Test project files are created.
  • Parsed: Source files are parsed for routes/components.
  • Resolved: Cross-file/module relationships are resolved.
  • Validated: Test asserts correctness of detected artifacts.

Integrations Involved

This feature interacts with several DeepDoc sub-systems:

  • Document Generation Pipeline: Consumes detected endpoints/components for documentation (Pipeline: Document Generation & Planning).
  • Frontend & UI Integration: Vue SFC detection feeds into UI documentation (Frontend & UI Integration Overview, Vue Component Parsing and Integration).
  • Webhook Integrations: Some tests touch webhook handler detection (Integration: Webhook Handlers).
  • LLM Client: Test coverage ensures correct context for LLM-based documentation (Integration: Anthropic, Integration: Deepdoc Chat API, Integration: Deepdoc Embed API).

Configuration & Environment

  • DEFAULT_CONFIG (deepdoc/config.py): Used in tests to configure scanning behavior.
  • No additional environment variables or feature flags are directly referenced in the test or detection logic.

Edge Cases & Failure Modes

  • Commented-Out Routes: Falcon detection skips routes if the add_route() call is commented out.
  • Sync/Async Wrappers: Django detection unwraps handlers wrapped in sync_to_async, csrf_exempt, etc.
  • Nested Includes: Django detection recursively resolves nested include() calls, including those using settings-based prefixes.
  • Multi-File Mount Chains: Express detection resolves routers mounted across multiple files, ensuring correct path resolution.
  • Constant Prefixes: Falcon and Django detection resolve path prefixes defined as constants or in settings files.
  • Missing Files/Imports: If a referenced handler or module cannot be resolved, the system falls back to the current file or skips the endpoint.
  • Cyclic Imports: The resolver tracks visited files to avoid infinite recursion.

If a project's import graph is malformed or contains ambiguous handler references, some endpoints may be missed or incorrectly attributed.
